## Dependency / ordering notes
- **Task 1 (lifts) before Task 3** — the actions' `run`s call the lifted handles; doing the lifts first means Task 3 only *declares*, never *duplicates*. **Task 2 (refs + constant) before Task 3** — the `run`s read the refs and the boundary test reads the constant.
- **Task 4 (boundary) and Task 5 (dedup + degradation) depend on Task 3** being green. **Task 6 (live)** depends on Task 3. **Task 7 is the final gate.**
- **CLE-05 interaction:** the two list-activate actions, `createSectorList`, `editPlan`, `regenerateScript`, `editScript`, `markRoleObsolete`, `rowEnrich`, `rowFindMobile`, `bulkFindMobile` are `confirm:"risky"` and `bookMeeting` is `outbound:true`+`confirm:"always"` → they reach CLE-05's confirm card before `run` (the credit/outbound ones show the "Uses credits"/"Sends externally" badge, CLE-05 §5). `selectProspect`/`setFromNumber`/`byDayView`/`sortQueue`/`writeEmailDraft` are `confirm:"never"` → CLE-03 runs them immediately. CLE-09 renders no card itself; it only declares the metadata that drives the gate.
- **The human-bound boundary (design §2) is enforced by omission + the Task 4 disjointness test** — this is the one design choice to foreground in the final report. Live WebRTC (dial/hang-up/voicemail-drop), in-call disposition (the 6 outcomes), call-again/skip, and buying a number (real money, admin-only) are **never declared**; the agent PREPARES the call and the human PLACES/DISPOSITIONS it and BUYS numbers (README §2). It changes no CLE contract.
- **CLE-11 (audit/undo):** these actions declare `reversible`/`outbound` honestly (list re-activate, plan re-PATCH, role-flag clear; `bookMeeting` outbound) but the undo *mechanism* + `tool_call_events` logging + the **cancellable-invite window** for `bookMeeting` is CLE-11 — out of scope here.
- **CLE-13 (send-guardrail hardening):** `bookMeeting`'s invite routes through `decideAction` (outbound → confirm) and the existing `/api/meetings/book` guardrails; sending-identity/opt-out/TZ-window hardening is CLE-13, not CLE-09.
- **Buy-number alternative (documented, not built):** if Martin prefers the README money *floor* over the human-bound choice, declaring `callMode.buyNumber` as `mutating:true, outbound:false, cost:"money", confirm:"always"`, `surfaces:["call-mode"]`, plus an admin-role guard, is an additive follow-up — but v1 ships it **excluded** (requirements §3, design §2/§7).
- **Bulk/cross-page ops stay headless** (README §3.6): "enrich every contact in France", "build a TAM", "find mobiles for the whole CRM" route to headless tools, not these page actions. `bulkFindMobile` is the one bulk page action and is scoped to the **loaded queue's** missing-mobile rows (the set the header bulk button targets), not the whole CRM. The CLE-04 prompt heuristic already teaches this; CLE-09 adds nothing for it.
