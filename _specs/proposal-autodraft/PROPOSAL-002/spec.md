# PROPOSAL-002: Auto-draft fill from the info base (DOCX)

Builds on PROPOSAL-001 (a `mapped` template with a confirmed component map).
Turns "Elevay understands the template" into "Elevay drafts the proposal."

## Requirements (GIVEN/WHEN/THEN)

### Feature 1: Resolve field components from structured data
**AC1.1** GIVEN a mapped template and a deal, WHEN fill runs, THEN every `field`
component is resolved from its `dataKey` against the deal/company/contact/seller/date
(deal.amount via `getDealAmountDisplay().total` + `formatDealAmount`, never a raw sum).
**AC1.2** GIVEN a `dataKey` whose source value is missing, THEN the field resolves to
an empty string (never the literal "undefined").

### Feature 2: Generate section components from the info base
**AC2.1** GIVEN `section` components, WHEN fill runs, THEN each section's prose is
generated by one grounded LLM call using the same info base as `draft-proposal`
(deal, company, stakeholders, conversation activities, notes, knowledge).
**AC2.2** GIVEN no LLM model is configured AND there are sections, THEN fill throws
`FillUnavailable('missing_required_data')` and persists nothing (no half-draft).

### Feature 3: Persist the filled proposal
**AC3.1** WHEN fill succeeds, THEN a `proposals` row (status='filled') + one
`proposal_components` row per component (with resolved/generated `content`, order)
are written, tenant-scoped, linked to the template + deal.

### Feature 4: Assemble the filled .docx (fidelity-preserving)
**AC4.1** GIVEN a filled proposal, WHEN the user downloads, THEN a .docx is produced
from the ORIGINAL template bytes with each anchored section's body replaced by the
generated content and each anchored field's value substituted, preserving the
original's other parts (styles.xml, headings, tables) by copying every other zip
entry unchanged.
**AC4.2** GIVEN a component whose anchor cannot be located in the document, THEN it is
reported as `unplaced` (never silently dropped, never mis-placed).

### Feature 5: Surfaces
**AC5.1** `POST /api/proposals/templates/[id]/fill { dealId }` runs the fill and
returns the proposal + components; `GET /api/proposals/[proposalId]` returns detail;
`GET /api/proposals/[proposalId]/download` streams the filled .docx. All tenant-scoped.
**AC5.2** A `fillProposal` chat tool and a "Fill from a deal" action on the proposals
page trigger the same path.

### Edge cases
- Template not `mapped` → 409 `template_not_mapped`.
- Deal not in tenant → 404.
- Field-only template (no sections) → fills without an LLM call.
- Section content with blank lines → one .docx paragraph per non-empty line.

## Design

- **Schema (`schema/proposals.ts` + migration 0060):** `proposals` (id, tenantId,
  templateId, dealId, createdByUserId, status, outputStorageRef, timestamps,
  deletedAt) and `proposal_components` (id, tenantId, proposalId, componentId, kind,
  label, placeholderToken, dataKey, content, source jsonb, confidence, order). The
  `source`/`confidence` columns are written by PROPOSAL-003 (trust); nullable now.
- **`lib/proposals/fill.ts`:** `resolveFieldValue(dataKey, ctx)`,
  `generateSections(sections, context, tenantId)` (one `tracedGenerateObject`,
  agentId `skill-proposal-fill-sections`), `buildProposalFill(templateId, dealId,
  {tenantId, userId})` orchestration; `FillUnavailable` error for clean abstention.
- **`lib/proposals/ooxml.ts` (write side):** `readAllZipEntries`, `writeZip` (STORE),
  `assembleFilledDocx(originalBytes, components, contentById)` → heading-anchored
  paragraph replacement, re-zipping with every other entry untouched. Limitations
  (SI-3): inline fields without an anchor and table-internal fills are `unplaced`;
  replaced paragraphs inherit the first replaced paragraph's pPr/rPr.
- **Skill `proposal-fill`** wraps `buildProposalFill`. **Routes** call the lib
  directly (auth-derived tenant/user). **Download** loads template bytes via
  `getProposalStorage().get`.
- **Reuse:** `draft-proposal`'s retrieval helpers + `getDealAmountDisplay`.
