## Dependency / ordering notes
- **Leaf-first:** Task 1 (migration/schema) + Task 2 (config) + Task 3 (snapshot union) before the helpers. Task 4 (`logPageActionCall`) + Task 6 (`outbound-hold.ts`) are independent and can be parallel. Task 5 (PAR reverse arm) needs Task 3; Task 7 (outbound reverse arm) needs Tasks 3+6. Task 8 (worker) needs Task 6. Task 9 (route + client) needs Task 4. Task 10 (undo tool + cancel route) needs Tasks 5+7. Task 11 (prompt) is independent after CLE-04's `<page_actions>` block exists. Task 12 (contract amendment) before merge. Task 13 is the gate.
- **CLE-10 interaction:** CLE-11 reads `decideAction`'s **output** (`execute` for outbound triggers the hold; `confirm` keeps the card path) — it does **not** change `decideAction`. If CLE-16 later makes outbound `execute` under autonomy, the hold is already the safety net (the audit's "fenêtre d'undo" thesis).
- **CLE-05/CLE-15 interaction:** CLE-11 ships the cancel **path** (`/api/outbound/[id]/cancel`) + the chat `undoLastAction` surface; a designed inline "Undo" toast is CLE-05/CLE-15 polish. CLE-11 does not depend on it.
- **CLE-13 interaction:** CLE-11 composes with the existing send guardrails and must not move them; CLE-13 hardens those guardrails (enforceSendingIdentity, opt-out everywhere, TZ windows). Order-independent; CLE-11's tests assert the *current* guardrails still fire at release.
- **Backwards-safety gate:** the whole feature is inert at `outboundUndoWindowSeconds = 0` (the default). Merging CLE-11 changes no live behaviour until a tenant opts into a window — the M2-checkpoint-safe posture (requirements §5).
