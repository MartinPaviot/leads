Transcript live

LIVE
api/calls/twiml + api/calls/transcription

Transcription native Twilio avec moteur Deepgram Nova-3 : Twilio héberge le stream et POST les chunks finals vers notre webhook. Pas de serveur WebSocket à tenir.

Deepgram Nova-3 moteur, via Twilio · fr-FR · both_tracks Twilio Transcription <Start><Transcription> · finals → calls.transcript → SSE Bridge auto-hébergé deepgram-bridge.ts · codé mais non utilisé
Nuance utile : Deepgram Nova-3 est le moteur de transcription (paramétré dans le TwiML), mais c'est l'intégration Twilio qui l'embarque. Notre DEEPGRAM_API_KEY en prod n'est donc pas consommée par ce chemin. Le seul mort ici est notre bridge WS maison.

Coaching live

LIVE
api/calls/transcription · coaching-classifier.ts

Sur chaque réplique prospect : préfiltre d'objection + debounce DB, puis classification → carte de réponse.

Claude Haiku 4.5 clé présente · classifyObjection → calls.coachingCards
Modèle réel = claude-haiku-4-5 (pas Sonnet). Gaté par VOICE_COACHING_LIVE + clé Anthropic. Debounce 5 s, anti-répétition de classe 60 s.

Analyse du transcript (post-appel)

LIVE (prod)
inngest/calls-post-process.ts

Webhook recording-status → event calls/post-process → extraction structurée. Inngest configuré en prod → s'exécute réellement.

Claude Sonnet 4.6 clé en prod · generateObject(callNotesSchema) OpenAI gpt-4o-mini clé en prod · fallback Deepgram batch fallback transcript vide NON implémenté
Seul trou : si le transcript est vide (stream raté), le code ne lance pas de transcription batch sur l'enregistrement — il classe en no_answer. Le reste de l'extraction est branché et tourne en prod.
