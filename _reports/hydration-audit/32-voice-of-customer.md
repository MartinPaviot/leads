# 32 — voice-of-customer (`/voice-of-customer`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is wired to a real, fully tenant-scoped data source: GET /api/voice-of-customer queries activities, notes, companies, and contacts all filtered by authCtx.tenantId (and isNull(deletedAt)), then runs an LLM (claude-sonnet-4-6) to extract themes/mentions. Loading and empty states are present, so it clears the H4/H3 bar — not unwired and not mocked. It falls short of the Home reference bar (H1) on two counts: (1) elements do not degrade independently or distinguish error from empty — any failure (no LLM key, 500 parse error, swallowed fetch throw) collapses into the generic "No customer insights yet" empty state, and (2) totalInteractions is omitted on the non-success response paths so the subtitle understates to 0. Content is LLM-generated rather than deterministic aggregation, so quotes/attribution carry paraphrase and mismatch risk. Net: H2 (partial).

Entrée : `app/apps/web/src/app/(dashboard)/voice-of-customer/page.tsx`.

## Pires défauts

1. Error/empty conflation: a 500 from LLM parse failure (route.ts:171), missing LLM key (route.ts:113), and a swallowed network throw (page.tsx:47-48) all render the same 'No customer insights yet' empty state (page.tsx:86-91) — failures are invisible to the user.
2. totalInteractions subtitle understates: it is only returned on the LLM success path (route.ts:169); the no-key and parse-error responses omit it, so the header reads '0 interactions' even when tenant interactions exist (page.tsx:82, route.ts:105/113).
3. Mention attribution is re-emitted by the LLM rather than joined by id (route.ts:129-134, prompt asks for 'exact quote or close paraphrase' at :130); company/contact/quote per mention can be hallucinated or mismatched against the real source rows shown at page.tsx:195-202.

## Résolution (P1 32 — fixed)

- **Defect #1 (error/empty conflation):** added a `loadError` state to the page; the fetch was refactored into a `load` callback that sets it on `!res.ok` (500 LLM parse failure) and on a thrown fetch, reset on retry. The body now renders `<EmptyState variant="error" title="Couldn't analyze your conversations" actionLabel="Retry" onAction={load}>` BEFORE the `themes.length === 0` empty state. A failure no longer reads as "no insights".
- **Defect #2 (subtitle understates):** the route now returns `totalInteractions: contentItems.length` on the no-LLM-key path (route.ts) and on the parse-error 500 path, and `0` explicitly on the genuine no-interactions path. With no key the subtitle now reads "0 themes from N interactions" (we have data, just couldn't analyze it) instead of "0 interactions".
- **Defect #3 (LLM-paraphrased / re-emitted attribution):** NOT addressed here — out of scope for a hydration error-state pass. The fix is a grounding redesign (join mentions back to real source rows by id rather than trusting the model to re-emit company/contact/quote). Flagged as a follow-up; conflating it with the error-state fix would be the "insane refactor" we avoid. The no-key path being a 200 (not 500) is left as-is: it's a deploy-config state (key is set in prod), and the corrected subtitle already signals "data present, not analyzed".

Verdict after fix: **H1** for the load/empty/error distinction and the subtitle count. The attribution-fidelity gap (#3) remains a documented follow-up. No route/page test harness; route change is purely additive (an existing field on more response paths); tsc clean.
