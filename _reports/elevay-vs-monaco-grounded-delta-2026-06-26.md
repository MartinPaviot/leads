# Elevay vs Monaco — grounded delta + first spec

> **What this is.** The Monaco UI teardown (`app/apps/web/_specs/MONACO-UI-TEARDOWN.md`) produced a 10-item build-list from Monaco's pixels. This doc grounds each item against Elevay's **actual code** (file:line), so the backlog reflects the real delta, not a competitor's surface. Date: 2026-06-26.
> **Method.** 10 grounding agents (one per build-list item) over `app/apps/web/src`, then synthesis. Two agents hit a flaky StructuredOutput cap per run, so this is a deterministic merge of two runs (run A: items 1-7,10; run B: items 8,9). Every claim is anchored to a file:line; "absent" is backed by the negative search.
> **Headline correction.** The autopilot is NOT missing — it lives on `main` behind `DAILY_AUTOPILOT_ENABLED` (default OFF): `inngest/daily-autopilot.ts` + `lib/autopilot/*` (19 files: run/budget/select/flag/candidates/enroll/prepare). Of 10 items: **1 present, 9 partial, 0 absent.** The engine is built and idle; the work is the guardrails that make flipping the flag on safe.
> **Branch caveat (verified 2026-06-26).** `lib/autopilot/*` and `inngest/daily-autopilot.ts` are on `main` but **NOT** on the current branch `feat/hydration-fidelity` (cut from an older main) — so anything built on the autopilot must branch off `main`. The grounding agents ran in the `feat/hydration-fidelity` working tree, so their `lib/autopilot/*` internal line numbers are reconstructed, not read — re-confirm at build time. The non-autopilot anchors the spec leans on were independently verified on `main`: `inngest/sequence-cron.ts:35-36`, `lib/analytics/optimizer/db-review.ts:178`+`183-184`, `inngest/daily-autopilot.ts:57-61`, `db/schema/outbound.ts:23`+`496-497` (`paused_at` already present).

## Throughline & recommended build order
### The engine is built and idle — ship the circuit-breakers, then flip the switch

Elevay is not missing the Monaco autopilot. It already has it. The full daily-autopilot loop lives on `main` behind `DAILY_AUTOPILOT_ENABLED` (default OFF): `inngest/daily-autopilot.ts` (weekday `0 7 * * 1-5` cron, flag-gated, tenant-scoped `countEnrolledToday`) drives `lib/autopilot/run.ts` — warmup-aware budget → `loadCandidates` (top `companies.priority_score`, targeted/non-excluded) → `selectProspects` → per-prospect `prepare` + `enroll`, fault-isolated, every send still passing `evaluateSend` at transport. So the question is not "can Elevay do what Monaco does" — it is "what has to be true before we set the flag to `1`."

Two of the ten items are the literal guardrails that make flipping it on **safe**, and both are *partially* built — the rails exist, the autonomous action is deliberately disabled:

- **#1 dead-channel/dead-sequence auto-pause is THE circuit-breaker.** This is exactly the Monaco failure the teardown caught: an autonomous sender firing 100/day into a 0-reply, 0-meeting channel with nothing stopping it. Elevay has every piece *except the autonomous pause*: regression detection (`lib/analytics/alerts/detect.ts:55`), a "pause" proposal type whose mutation is deliberately stubbed (`lib/analytics/optimizer/db-review.ts:178-185` — `dbIsAutonomous()` hardcoded `false`, `dbApplyChange` throws), the `"paused"` value in the `sequence_status` enum (`db/schema/outbound.ts:20-24`), and the send gate that makes a pause real (`inngest/sequence-cron.ts:31-40` halts due steps when `sequences.status != 'active'`; `daily-autopilot.ts getActiveSequenceId` only enrolls into an `active` sequence). The missing 20% is a meetings/reply outcome metric and the wired pause action. **Ship #1 first — it is the prerequisite that makes autonomous enrollment reversible.**

- **#2 discriminating (forced-distribution) score is the approval-trust guardrail.** Grade+heat, per-criterion weights, and source/`detectedAt` provenance all already exist (`lib/scoring/scoring.ts:9-34`, `lib/util/ui-utils.ts:65-121`, `lib/scoring/score-account.ts:9-49`). The genuinely-absent piece is the *forced distribution* — grades today are absolute cutoffs, so a loose ICP makes everything read "A/Burning" and the founder loses the signal that tells them which auto-enrollments to trust. Note the nuance: autopilot *selection* ranks on the continuous `priority_score` (`lib/autopilot/select.ts`), so the curve is not required for correct picking — it is required for an **honest approval UI** once volume ramps. Guarded pre-launch item, not a hard selection gate.

The rest split into operational-safety-prereqs vs polish:

- **#5 autopilot cockpit** — the veto/bulk-approve half is already shipped (`sequences/review/page.tsx`), but the **observability + one-click Pause half is a prerequisite**: today the only pause levers are the global env flag and an unsurfaced `dailyAutopilotBudget=0`. Flipping on without a per-channel "sent today" ledger and a tenant Pause button is operationally blind.
- **#7 cross-channel inbox** — the review-before-send composer (the HITL guardrail) is present for email. But memory says **LinkedIn is the primary channel**, and there is no LinkedIn-inbound→`activities` capture path (`lib/sequence-dispatch/linkedin-adapter.ts:33-34` is a stub). Once autopilot sends on LinkedIn, replies arrive on LinkedIn and the inbox silently drops them. **The ingestion half is a prerequisite for the primary channel** (don't lose replies); the badge/copy/attachment work is polish.
- **#10 deliverability** — the safety floor (warmup ramp + bounce auto-throttle + 6h health auto-pause) already exists and is wired, so this is **not** a hard prereq. The cheap near-prereq before scaling volume is DNS/DMARC *persistence + enforcement* (autonomous sending without enforced DMARC is the real reputation risk). Land-detection + LinkedIn fallback are post-launch differentiators.

Independent polish (improve trust/throughput, do not gate the flip): **#3** ICP matrix (already exceeds Monaco; the one real item is the latent jsonb-merge data-loss bug at `inngest/custom-signal-backfill.ts:145`), **#4** booking loop (already complete and tested — better than Monaco, which drops the time; only surfacing polish remains), and the convenience half of **#6** (bulk enroll/suppress/export — though the cheap **N>500 confirm** is worth doing before any human hand-enrolls large lists).

### Recommended build order
1. **#1 dead-sequence auto-pause** (P0, M) — the circuit-breaker. Hard prerequisite to flipping the flag.
2. **#2 forced grade distribution** (P0, M) — approval-trust; guarded, ship before volume.
3. **#5 cockpit Pause + per-channel sent ledger** (P0-operational, M) — the kill-switch + visibility; veto half already shipped.
4. **#7 LinkedIn-inbound ingestion** (P0-for-LinkedIn, L) — only the ingestion path; required before flipping on the primary channel.
5. **#10 DNS/DMARC persistence + enforcement** (P1, S) — cheap reputation guardrail before scaling volume.
6. **#6 N>500 confirm** (P1, S) — cheap human-enroll guardrail; defer enroll/suppress/export (M).
7. **#3 jsonb-merge fix + matrix polish** (P2, S then M) — fix the data-loss bug first; presets/re-enrich/bulk-enroll after.
8. **#4 booking-loop surfacing** (P2, S) — promote the existing "Book {phrase}" out of the overflow menu; thread onto home cards.

Flip `DAILY_AUTOPILOT_ENABLED=1` only after 1, 3, and (for LinkedIn) 4 are live and smoke-tested; ship 2 alongside for honest UI; 10/6 before scaling volume; 3/4 anytime.

### Addendum — items #8 and #9 (from the second grounding run)
These two are reporting-fidelity, not send-safety, so they sit after the flip in the order above — **but two real bugs they surfaced should be fixed regardless of sequencing** (both in `app/api/dashboard/summary/route.ts`): the pipeline-value counter sums **all** deals incl. closed (`summary/route.ts:214`) while `/api/pipeline` uses open-only (divergence = the Monaco "closed 40 vs Pipeline 0" anti-pattern, in our own code), `activeDeals` counts **won+lost** (`summary/route.ts:215`), and the enrollments query is **missing `tenantId`** (`summary/route.ts:84-98`) — a cross-tenant leak. #9 (sequence funnel + meetingRate) is read-only and renders 0 until enrollments flow; its durable value is meeting attribution, a precursor to any "promote what books meetings" optimizer. Full grounding for both below.

## Per-item grounding (all 10)
### 1. Dead-channel / dead-sequence auto-detect → auto-pause
**Status: partial.** The rails are built; the autonomous pause is deliberately disabled. Regression detection exists (`lib/analytics/alerts/detect.ts:55-78`, baseline-relative magnitude, not an absolute floor, alert-only). A "pause" proposal type exists but is observe-only: `lib/analytics/optimizer/db-review.ts:178-180` hardcodes `dbIsAutonomous()→false` and `dbApplyChange` (`:183-185`) throws `"optimizer auto-apply is not implemented"`; the weekly cron (`inngest/weekly-optimizer.ts`) only persists `optimizer_proposal` + notifies a human. Mailbox-level auto-pause does exist (`inngest/deliverability-monitor.ts:51-73`) but pauses *mailboxes* on bounce/complaint, not outcome-dead sequences. The plumbing that makes a pause real is in place: `sequence_status` enum includes `"paused"` (`db/schema/outbound.ts:20-24`) and `inngest/sequence-cron.ts:31-40` gates due-step sends on `sequences.status='active'`, while `daily-autopilot.ts getActiveSequenceId` only enrolls into an `active` sequence — so flipping `status='paused'` halts both delivery and autopilot targeting.

**Delta.** Add the missing outcome dimension + the wired pause: (1) a pure detector that flags a sequence DEAD when, over a window, `sent >= MIN_SAMPLE AND replyRate < REPLY_FLOOR AND positiveReplies == 0 AND meetingsBooked == 0` — reusing the analytics route's `sequenceEnrollments → outboundEmails` join (`api/sequences/[id]/analytics/route.ts:76-130`) plus a meetings count from `pipeline_events` stage `meeting_booked` (`lib/analytics/pipeline-tracker.ts:16`, table `campaign.ts:194-224`, joined by `enrollmentId`/`contactId`); (2) a tenant-scoped pause mutation (`UPDATE sequences SET status='paused', paused_reason, paused_by, paused_at WHERE id=… AND tenant_id=…`); (3) a small additive migration for the audit columns; (4) a flag-gated cron + human notification mirroring `deliverability-monitor.ts:62-71`.

**Effort: M.** Detection scaffold, pause proposal type, the `"paused"` enum, the cron gate, and the notification pattern all already exist. Net-new is the outcome metric (meetings join + absolute reply floor), the wired tenant-scoped mutation, the audit-column migration, and regression tests. Not S (new detector + mutation + migration + scoping); not L (no new engine).

**Priority: P0 — hard prerequisite.** This is the exact Monaco failure mode the teardown flags. Without it, turning autopilot on reproduces Monaco: 100/day into a 0-reply/0-meeting channel with no circuit-breaker. Ship before flipping `DAILY_AUTOPILOT_ENABLED`.

**Risks.** Tenant-scoping the UPDATE (the optimizer's target is a `campaignId` — don't trust it blindly). False positives on thin data → require `MIN_SAMPLE` + treat below-sample as `insufficient_data`, never pause. Reversibility → must be human-resumable with a re-pause cooldown so it doesn't immediately re-pause a resumed sequence. Channel-vs-sequence → `"paused"` is sequence-level; a sequence with one dead and one healthy channel should NOT be auto-paused in v1 (log a partial-dead advisory instead). Migration → journal frozen at idx 12; use `db:push` (localdev) / `DATABASE_URL_OWNER` (prod).

### 2. Two-axis explainable scoring (grade+heat) with weights, source, detectedAt, forced distribution
**Status: partial.** The two-axis model is built and richer than Monaco's: `lib/scoring/scoring.ts:9-34` (GRADE_THRESHOLDS A+…F each mapped to a HeatLevel, `getGrade`), surfaced via `lib/util/ui-utils.ts:65-121` (`displayScore`). Numeric `companies.score` (`db/schema/core.ts:65`) and composite `priority_score` (`core.ts:92`, recomputed by `inngest/signal-score-daily.ts`) both exist. Per-criterion weights + explainability are real (`lib/scoring/score-account.ts:9-49`, `Contribution[]`); provenance (source + `detectedAt`) renders in the SignalChip popover (`components/signal-chip.tsx:225-321`). The one genuinely-absent capability: a **forced/relative grade distribution** — grep for percentile/quantile/forced/curve returned no cohort-relative grading; grades are purely absolute cutoffs.

**Delta.** (A) the net-new piece: a pure `computeForcedGrades(scores[], targetQuantiles)` that buckets a tenant's scored companies by rank-percentile (A=top ~5%, etc.), called inside `signal-score-daily.ts` after scores compute, persisting `companies.grade` + `companies.grade_basis` so the table/filters read the curved grade; keep an env/setting toggle so absolute grading stays default until validated. (B) optional polish: a single ScoreBreakdown component merging the existing `Contribution[]` + SignalChip provenance.

**Effort: M.** Grade+heat, weights, provenance all exist. New logic = one pure module + a small migration (`grade`/`grade_basis`) + wiring into the recompute cron + reading the curved grade. (B) is S.

**Priority: P0 — guarded pre-launch (approval trust).** Autopilot *selection* ranks on the continuous `priority_score` (`lib/autopilot/select.ts`), so the curve is not required for correct picking — it is required for honest UI once autopilot enrolls at volume (a loose ICP makes everything read "A/Burning"). Ship (A) guarded before launch; (B) post-launch.

**Risks.** Distribution MUST be per-tenant over eligible (non-excluded, non-deleted) companies — reuse the tenant boundary in `signal-score-daily.ts`; a global quantile leaks cross-tenant. Route the curved grade through the single `getGrade`/`ui-utils` path so a third grade source doesn't fragment truth (`scoring.ts` vs `score-account.tierFor` vs hot-to-call bands). Persist `grade_basis` (cohort size + computedAt) so the UI can explain why a letter moved. Migration via `db:push`/`DATABASE_URL_OWNER`.
