## Bruit console PostHog/CSP — cause racine : env e2e (PAS un bug produit)

Sur toutes les pages, la console montre des erreurs CSP PostHog
(`us.i.posthog.com` / `us-assets.i.posthog.com` bloqués par une CSP qui
n'autorise que `eu.i.posthog.com`).

Cause racine confirmée : **le serveur testé sur :3000 est le webServer e2e**
(`playwright.config.ts`, `reuseExistingServer: true`), qui force
`NEXT_PUBLIC_POSTHOG_HOST = https://us.i.posthog.com` + clé `phc_e2e_test`.
La CSP de l'app est volontairement **EU-only** (souveraineté ; figée par
`src/__tests__/csp-allowlist.test.ts`). US host + EU CSP → bloqué.

Le `.env.local` réel est cohérent (`NEXT_PUBLIC_POSTHOG_HOST=https://eu.i.posthog.com`,
clé EU). Donc en dev normal (`npm run dev`) et en prod, PostHog n'est PAS bloqué.

→ **Rien à corriger côté produit.** Au plus, on pourrait aligner le
`playwright.config.ts` sur l'hôte EU pour supprimer le bruit pendant les e2e,
mais les tests interceptent l'appel PostHog (no-op sans vraie clé) donc c'est sans
effet fonctionnel.

## Contexte d'exécution (important)

Un **agent concurrent fait tourner des tests Playwright e2e** dans ce repo
(webServer e2e sur :3000, specs `tests/e2e/*`, clé `phc_e2e_test`, tenant
"E2E Test Workspace"). Conséquences observées pendant ce balayage :
- fermetures intermittentes du navigateur (collision Playwright — "one browser
  at a time"),
- `tsc` TS6053 transitoire (`verify-settings-headers.spec.ts` absent par moments),
- données = tenant de test, pas le tenant Pilae réel.
Le balayage a donc validé le tenant de test ; un re-run sur le tenant réel
(`47dca783`, admin) reste à faire pour les actions admin-gated (save Settings,
etc.).
